\section{\IfLanguageName{dutch}{Wat is CA Gen?}{What is CA Gen?}}%
\label{sec:Wat is CA Gen?}
 CA Gen is een Computer Aided Software Engineering (CASE)-tool en een vierde generatie ontwikkelomgeving (4GL), oorspronkelijk ontwikkeld in 1987. CA Gen is doorheen de jaren onder verschillende namen bekend geweest, waaronder IEF (Information Engineering Facility), Composer by IEF, Composer, COOL:Gen, Advantage:Gen en AllFusion Gen. Met deze CASE-tool kunnen ontwikkelaars op modelgedreven wijze applicaties ontwerpen via grafische modellering en gestructureerde specificaties, zonder rechtstreeks traditionele programmeersyntax te schrijven. Op basis van deze modellen genereert CA Gen automatisch broncode in verschillende programmeertalen, zoals C en COBOL, en voor diverse platformen. CA Gen ondersteunt onder meer component-based software engineering en verschillende architectuurvormen, waaronder client-server-, batch-, online- en webapplicaties. De tool combineert functioneel ontwerp, datamodellering, technische implementatie en deployment binnen één geïntegreerde ontwikkelomgeving. Alle modellen en gegenereerde artefacten worden opgeslagen in één of meerdere zogenaamde encyclopedieën op het mainframe. Deze encyclopedieën maken gebruik van een DB2-database voor de opslag van metadata en modelinformatie. Binnen ArcelorMittal bevat de encyclopedie verschillende modellen, waarbij elk model een business- of technisch model representeert. De businessmodellen bevatten de logica en specificaties die nodig zijn om een applicatie te genereren, implementeren en testen. 
 
 \section{\IfLanguageName{dutch}{Host encyclopedie}{Host Encyclopedia}}%
 \label{sec:Host Encyclopedie}
 Een Gen-encyclopedie is een centrale opslagplaats van objecten waarmee meerdere gebruikers tegelijkertijd aan meerdere ontwikkelingsprojecten kunnen werken. CA Gen biedt de volgende twee types encyclopedie aan:
 \begin{itemize}
     \item Een host encyclopedie die draait op z/OS gebruike makende van TSO
     \item Een Client Server Encyclopedie (CSE) that draait op een netwerk gebruik makende van één of meerdere werkstations
 \end{itemize}
 De host encyclopedie is een DB2 database dat informatie opslaat dat de Gen Host componenten creëren op meerdere werkstations. Met deze encyclopedie kan je alle kritieke taken controleren vanuit een centrale locatie. Met deze encyclopedie kan heel veel gaan doen, maar in deze context is het belangrijkste de “Check in”, ofwel het uploaden van een model, en de “Check out” ofwel het downloaden van een model.
 
 \section{\IfLanguageName{dutch}{Modellen}{Models}}%
 \label{sec:Modellen}
 CA Gen werkt dus fundamenteel anders dan klassieke programmeertalen. In plaats van rechtstreeks code te schrijven, bouw je een applicatie op via modellen. Die modellen vormen de kern van hoe applicaties worden ontworpen, gegenereerd en onderhouden. CA Gen-modellen zijn abstracte beschrijvingen van een applicatie waarin zowel de functionele logica, datadefinities als gedrag worden vastgelegd, zonder dat er onmiddellijk programmeercode geschreven wordt. Je kan ze zien als een blauwdruk van een applicatie waaruit automatisch code wordt gegenereerd. Volgende soorten modellen bestaan:
 \begin{itemize}
     \item Business models (functionele modellen). 
     Dit zijn de belangrijkste modellen. Deze gaan beschrijven wat de applicatie gaat doen, welke logica uitgevoerd wordt en hoe data verwerkt zal worden zoals bijvoorbeeld berekeningen. Dit gebeurt in een gestructureerde pseudocode waarover later meer.
     \item Data models (datamodellen).
     Deze modellen definiëren entiteiten (bijvoorbeel een klant), attributen (velden) of relaties tussen entiteiten. Dit is vergelijkbaar met een ER-diagram. Deze modellen bepalen ook hoe data uiteindelijk in DB2 wordt opgeslagen.
     \item Action diagrams
     Een action diagram beschrijft de stap-voor-stap logica, hoe data verwerkt wordt en welke acties uitgevoerd worden. Dit ook in een pseudocode.
 \end{itemize}
 Het werken met deze modellen zorgt voor abstractie, je werkt op een hoger niveau dan code wat ervoor zorgt dat code kan gegenereerd worden in verschillende programmeertalen. Anderzijds is deze gegenereerde code wel heel moeilijk te verstaan zonder de modellen.
 
 \section{\IfLanguageName{dutch}{CASE migratie}{CASE migration}}%
 \label{sec:CASE}
 Zoals voorheen uitgelegd is CA Gen een CASE-tool. Een CASE migratie omvat het verplaatsen van softwaresystemen, ontwikkelomgevingen en de bijhorende data van verouderde (legacy) omgevingen naar een ander platform. Dit is een complex proces dat je toelaat om legacy-applicaties te upgraden, te herontwerpen of te migreren naar modernere technologieën, om de prestaties, onderhoudbaarheid of wendbaarheid te verbeteren. 
 Enkele belangrijke aspecten van een CASE-migratie:
 \begin{itemize}
     \item Migratietypen: Migratie kan bestaan uit het verplaatsen, transformeren of vervangen van infrastructuur of applicaties.
     \item Procesfasen: Een succesvolle migratie volgt doorgaans een gestructureerd proces van planning, continu of stapsgewijze uitvoering en validatie.
     \item Migratie kan de efficiëntie van je infrastructuur/applicaties verhogen en je kosten verlagen. Als je wil is AI-integratie ook mogelijk.
     \item Risicoreductie: Een incrementele aanpak, waarbij componenten in kleine stappen worden gemigreerd, zorgt dat het systeem operationeel blijft, waardoor de verstoring van de bedrijfsvoering tot een minimum wordt beperkt.
 \end{itemize}
 
 Veelvoorkomende migratie-uitdagingen:
 \begin{itemize}
     \item Complexiteit: Systemen die gebasseerd zijn op verouderde 4GL-talen vereisen vaak aanzienlijke herontwikkeling om over te stappen naar moderne architecturen.
     \item Compatibiliteit: Het waarborgen dat oude componenten tijdens de overgang blijven functioneren naast de nieuwe.
     \item Tekort aan vaardigheden: De behoefte aan expertise op het gebied van nieuwe frameworks en architecturen.
 \end{itemize}
 
 \section{\IfLanguageName{dutch}{CA Gen Views}{CA Gen Views}}%
 \label{sec:Views}
 Het begrip view is essentieel voor hoe data door de applicatie stroomt en hoe de database wordt aangesproken. Er wordt doorgaans een onderscheid gemaakt tussen data-views (in Action Diagrams) en UI-views (in de ontwikkelomgeving). Hieronder volgt een overzicht van de verschillende typen views die je in de logica tegenkomt:
 \newline
 
 \begin{easylist}
     § Functionele categorieën
     §§ Import Views: De inputvariabelen. Deze data wordt meegegeven aan het Action Block door de aanroepende logica.
     §§ Export Views: De outputvariabelen. Deze data word door het Action Block teruggegeven aan de aanroepende logica.
     §§ Local Views: Tijdelijke variabelen die alleen binnen het betreffende Action Block bestaan.
     \newline
     § Inhoudelijke categorieën
     §§ Entity Views: Deze zijn direct gekoppeld aan een Entity Type uit het datamodel. Ze worden gebruikt om gegevens uit de database te verwerken.
     §§ Work Views: Deze zijn gebasseerd op een Work Set. Dit zijn variabelen die niet in de database staan, zoals tellers, flags of tijdelijke berekeningen.
     §§ Entity Action Views: Een specifiek type view dat gebruikt wordt binnen de database-statements zoals READ, CREATE, UPDATE en DELETE. Ze dienen als buffer voor de database-operatie.
     \newline
     § Structurele categorieën
     §§ Group Views: Dit zijn “repeating views” of arrays. Een Group View bevat één of meer Entity- of Work Views en stelt de applicatie in staat om lijsten met records te verwerken.
     §§ Recursive Views: Hoewel minder vaak gebruikt, kunnen views recursief worden gedefinieerd voor boomstructuren of hiërarchische data.
 \end{easylist}
 
 \section{\IfLanguageName{dutch}{Syntax mapping}{Syntax mapping}}%
 \label{sec:Syntax}
 Zoals eerder vermeld maakt CA Gen gebruik van een vorm van gestructureerde pseudocode waarin de volledige applicatielogica wordt gedefinieerd. Deze pseudocode vormt de basis voor de automatische generatie van programmacode.
 
 In dit onderdeel wordt voor de meest voorkomende CA Gen-expressies en constructies toegelicht hoe deze systematisch kunnen worden vertaald naar equivalente structuren in PL/I. Hierbij wordt gefocust op het correct interpreteren van de onderliggende logica en het behouden van functionele equivalentie tussen de oorspronkelijke CA Gen-implementatie en de PL/I-code. Dit wordt gedemonstreerd aan de hand van enkele echte voorbeelden.
 
 \subsection{Variabelen}
 Variabele in CA Gen verschillen op een aantal belangrijke punten van variabelen in klassieke programmeertalen. Omdat CA Gen modelgedreven werkt, zijn variabelen sterk gekoppeld aan het datamodel en de context waarin ze gebruikt worden. In CA Gen zijn variabelen gegevenshouders die gebruikt worden binnen action diagrams en procedures om waarden tijdelijk of persisten op te slaan tijdens de uitvoering van een applicatie. Volgende soorten variabelen bestaan:
 \begin{itemize}
     \item De meest voorkomende variabelen zijn attribute-based variabelen, deze zijn afgeleid van een entiteit of datamodel en hebben een vaste structuur en datatype. Voorbeelden hiervan zijn 'Customer.Name' of 'Material.ID'. Deze variabelen zijn rechtstreeks gelinkt aan de database-structuur.
     \item Local variables: deze worden gebruikt binnen procedures en action diagrams. Variabelen van deze soort bestaan slechts tijdelijk (enkel tijdens de uitvoering) en worden bijvoorbeeld gebruikt voor berekeningen of tussenresultaten. De lokale variabelen worden gedefinieerd aan het begin van de action diagram onder 'IMPORTS'.
     \item Parameter-variabelen worden gebruikt als I/O voor procedure calls of module-interactie.
     \item CA Gen voorziet ook systeemvariabelen zoals statuscodes en foutcodes.
 \end{itemize}
 
 \subsection{SET...TO}
 In CA Gen gebeurt het toekennen van een waarde aan een variabele via de sleutelwoorden SET en TO. 
 De syntactische structuur die hierbij wordt gehanteerd is als volgt: SET \{variabelenaam\} TO \{waarde\}. Hierbij verwijst {variabelenaam} naar de doelvariabele waaraan een nieuwe waarde wordt toegekend, terwijl {waarde} een constante, een andere variabele of het resultaat van een expressie kan zijn. De PL/I equivalent van de SET-instructies uit figuur \ref{fig:Vb 1 CA Gen-code} zou er als volgt uitzien:
 \begin{verbatim}
     DCL 1 INT_PLN_PLNPRDO_PRODUCTIE_ORDER,
         2 ID_BS_PRD FIXED              BIN(31);
     DCL 1 INT_MAT_MATERIAALEENHEID,
         2 NUMMER FIXED                 BIN(31),
         2 CODERING_IDENTIFICATIENUMMER CHAR(7);
     DCL FLAG                           CHAR(8);
        
     INT_PLN_PLNPRDO_PRODUCTIE_ORDER.ID_BS_PRD = 0;
     INT_MAT_MATERIAALEENHEID.NUMMER = 0; 
     INT_MAT_MATERIAALEENHEID.CODERING_IDENTIFICATIENUMMER = '';
     FLAG = '';         
 \end{verbatim}

 \subsection{WHILE-loop}
 Een WHILE-loop in CA Gen wordt als volgt gedefinieerd: WHILE \{conditie\} ... REPEAT \{code\}. De lus gaat de code na de repeat uitvoeren tot de conditie niet meer geldig is. Nog een keyword dat in veel loops wordt gebruikt is 'ESCAPE', dit wil zeggen dat je uit de loop stapt en de pijl duidt aan tot welk niveau je terugkeert. De code uit figuur \ref{fig:Vb 2 CA Gen-code} zou worden vertaald naar: 
 
 \begin{verbatim}
     /* WHILE */
     DO WHILE (INT_AFF_WERKVELDEN.NR_VKM <= INT_AFF_WERKVELDEN.Q_VKM);
       /* REPEAT */
       GRL_TTIS = 1;
       INT_AFF_MATERIAALEENHEID.NR_BS_AFF    = INT_AFF_TTI.NR_BS;
       INT_AFF_MATERIAALEENHEID.NR_PO_BS_AFF = INT_AFF_TTI.NR_PO_BS;
       
       IF INT_AFF_MATERIAALEENHEID.FRM_AFD_PRD ^= “SID”
       THEN
         /* ESCAPE */
         LEAVE;
     END;
 \end{verbatim}

 \subsection{USE}
 Het USE-statement is een van de meest fundamentele commando's in een Action Diagram. Het stelt je in staat om de logica van het ene Action Block aan te roepen vanuit een ander. Het USE-statement fungeert als een subroutine-call. In plaats van alle logica in één groot blok te schrijven, splits je functionaliteiten op in kleinere, herbruikbare eenheden.
 \begin{itemize}
     \item Common Action Blocks (CABs): Logica die binnen de CA Gen-encyclopedie is gedefinieerd.
     \item External Action Blocks (EABs): Logica die buiten Gen is geschreven (bijvoorbeeld PL/I).
 \end{itemize}
 Een belangrijk concept bij een USE-statement is View Matching. Je kan niet zomaar variabelen overal gebruiken, je moet expliciet definiëren welke data “naar binnen” gaat en welke “naar buiten” komt.
 \begin{itemize}
     \item Import Matching: Je koppelt views uit het aanroepende blok aan de Import Views van het aangeroepen block.
     \item Export Matching: Je koppelt de Export Views aan het aangeroepen blok terug aan de views in het aanroepende blok.
 \end{itemize}
 Wat mag je niet doen met het USE-statement:
 \begin{itemize}
     \item Local Views: Deze kunnen niet gematcht worden met een ander blok.
     \item Recursive Calls: Ca Gen laat standaard geen directe recursie toe.
     \item Performance: Te veel geneste USE-statements.
 \end{itemize}
 De exitstate van een Action Block geeft aan of de blok juist is afgehandeld. Hiervoor is geen directe tegenhanger in PL/I.
 De code in figuur \ref{fig:Vb 3 CA Gen-code} zou er in PL/I zo uit zien:
 
 \begin{verbatim}
     
     CALL(SID_WERKVELD, GRI_PRDO);
     
 \end{verbatim}
